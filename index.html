Developer & Integration Guide

Onboarding resource for configuring client chatbots, webhook routers, n8n orchestrators, and CRM synchronization.

1. System Architecture Flow

WebsiteBot runs on a highly decoupled architecture. The frontend widget never communicates with databases or n8n directly, keeping API keys, secrets, and private webhook endpoints safe from client-side visibility.

1. Embedded Widget Loads on client site and queries Worker with unique Project UUID.
➔
2. Cloudflare Router Fetches styling & routes messages using Supabase metadata.
➔
3. n8n Orchestrator Triggers LLM agent, executes catalog searches, or maps to CRM.
➔
4. Client CRM Salesforce or HubSpot stores the captured lead data.

2. Client Project Modes

Clients and configurations can be run in two execution scopes:

  • Local Mode (LocalStorage Playground): Changes are stored in browser storage. Generates static integration snippets that have styling attributes hardcoded inline.
  • Cloud Mode (Supabase Connected): Changes save automatically to Supabase. Generates a dynamic integration snippet containing only a data-project-id UUID. Updates made in this console take effect on live sites immediately.

3. Webhook Router (Cloudflare Worker)

The global routing endpoint https://api.krutrimkarta.com/chat acts as a secure proxy:

GET /chat?projectId={UUID}  ➔ Fetches styling from Supabase
POST /chat                  ➔ Resolves private n8n URL and forwards payload

This allows your clients to move hosting or redeploy workflows without ever modifying the embed code pasted on their website.

4. n8n Pipeline Setup

To integrate an n8n workflow as the chat brain:

  1. Create a Webhook node listening to POST requests (copy this URL to the client's Webhook input in the dashboard).
  2. Create a Supabase node to query configuration columns matching the incoming projectId (e.g. loads system_prompt_extension, integrations).
  3. Create an AI Agent Node with your language model, appending the dynamic prompt extension from Supabase.

5. CRM Sync (HubSpot / Salesforce)

Avoid hardcoding API keys in n8n workflows. Store access details dynamically in Supabase:

  1. Under the client's database row, write details in the integrations cell: 
    {
  "crm": "hubspot",
  "crm_token": "pat-na1-xxxx-xxxx"
}
  2. In your n8n workflow, route requests based on the resolved integrations.crm value.
  3. Bind the CRM node auth tokens using the dynamic property expression: {{ $json.integrations.crm_token }}.

6. Lead Capture Data Path

Websitebot does not log or persist customer lead data in its central database in production. Captures run on a strict privacy route:

This isolation ensures customer privacy and guarantees that all contacts sit directly and exclusively in the client's CRM.

7. Configuration Versions & Rollbacks

To audit changes and recover from accidental configuration mistakes, WebsiteBot includes version tracking:

  • Automatic Snapshot: Edits trigger a snapshot of the old configuration state inside project_config_versions before overwriting it.
  • 60s Coalescing: To avoid database pollution during active customization (e.g. typing or color selecting), a new snapshot is logged only if at least 60 seconds have elapsed since the last log.
  • Retention Limit: History is capped at the 5 most recent versions per project.
  • Reversible Restore: Restoring a past version automatically saves the pre-restore state as a new entry, allowing you to undo rollbacks.

Database Setup Instructions

Create a new table in your Supabase database by running the SQL query below in your Supabase SQL Editor:

CREATE TABLE IF NOT EXISTS kartabot_projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL UNIQUE,
  bot_name TEXT NOT NULL DEFAULT 'KartaBot',
  avatar_url TEXT NOT NULL DEFAULT 'https://images.unsplash.com/photo-1618005182384-a83a8bd57fbe?w=100&auto=format&fit=crop&q=80',
  welcome_message TEXT NOT NULL DEFAULT 'Hello! Welcome to our website. How can I help you today? 😊',
  n8n_url TEXT,
  suggestions TEXT,
  theme TEXT NOT NULL DEFAULT 'light',
  color TEXT NOT NULL DEFAULT '#6366f1',
  position TEXT NOT NULL DEFAULT 'right',
  created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL,
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL
);

ALTER TABLE kartabot_projects ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Allow public read and write" ON kartabot_projects FOR ALL USING (true) WITH CHECK (true);

-- Create version history table
CREATE TABLE IF NOT EXISTS project_config_versions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES kartabot_projects(id) ON DELETE CASCADE,
  version_number INT NOT NULL,
  config JSONB NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_project_config_versions_project_id ON project_config_versions(project_id);

ALTER TABLE project_config_versions ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Allow public read and write" ON project_config_versions FOR ALL USING (true) WITH CHECK (true);

-- Trigger function to log old configuration state on update
CREATE OR REPLACE FUNCTION log_project_config_version()
RETURNS TRIGGER AS $$
DECLARE
  next_ver INT;
  last_created TIMESTAMP WITH TIME ZONE;
  old_conf JSONB;
  new_conf JSONB;
BEGIN
  old_conf := jsonb_build_object(
    'bot_name', OLD.bot_name,
    'avatar_url', OLD.avatar_url,
    'welcome_message', OLD.welcome_message,
    'n8n_url', OLD.n8n_url,
    'suggestions', OLD.suggestions,
    'theme', OLD.theme,
    'color', OLD.color,
    'position', OLD.position
  );

  new_conf := jsonb_build_object(
    'bot_name', NEW.bot_name,
    'avatar_url', NEW.avatar_url,
    'welcome_message', NEW.welcome_message,
    'n8n_url', NEW.n8n_url,
    'suggestions', NEW.suggestions,
    'theme', NEW.theme,
    'color', NEW.color,
    'position', NEW.position
  );

  IF old_conf IS DISTINCT FROM new_conf THEN
    SELECT max(created_at) INTO last_created 
    FROM project_config_versions 
    WHERE project_id = OLD.id;

    -- Coalesce version logging (only record if > 60s since last version)
    IF last_created IS NULL OR (EXTRACT(EPOCH FROM (now() - last_created)) > 60) THEN
      SELECT COALESCE(max(version_number), 0) + 1 INTO next_ver 
      FROM project_config_versions 
      WHERE project_id = OLD.id;

      INSERT INTO project_config_versions (project_id, version_number, config, created_at)
      VALUES (OLD.id, next_ver, old_conf, now());

      -- Retention Policy: Cap versions at 5 per project
      DELETE FROM project_config_versions
      WHERE id IN (
        SELECT id FROM project_config_versions
        WHERE project_id = OLD.id
        ORDER BY version_number DESC
        OFFSET 5
      );
    END IF;
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_log_project_config_version ON kartabot_projects;
CREATE TRIGGER trg_log_project_config_version
BEFORE UPDATE ON kartabot_projects
FOR EACH ROW
EXECUTE FUNCTION log_project_config_version();

Ensure Row Level Security (RLS) is configured to allow public reads and writes as defined by the policy above for testing.
